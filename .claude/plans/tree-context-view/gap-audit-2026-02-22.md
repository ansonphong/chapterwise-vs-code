## Executive Summary

The staged plans are broadly directionally correct, but implementation is still largely pre-plan. Most Stage 1-7 deliverables are not present in code. There are also several plan-quality gaps that would cause regressions or incomplete behavior even if implemented as written.

Highest-risk gaps are:

1. Unified ordering migration is not implemented and plan does not fully cover all dependent code paths.
2. New context-menu work is scoped to `chapterwiseCodexNavigator`, while default display mode is `stacked`, which uses different views.
3. Planned reload pattern (`reloadTreeIndex`) does not consistently regenerate index cache before reload.
4. Plan snippets introduce potential security and UX issues (path traversal checks missing, keybinding conflicts, batch command prompt loops).

## Recommended Plan Corrections Before Implementation

1. Expand menu/keybinding scope strategy to include stacked view IDs (`chapterwiseCodexIndex0-7`, `chapterwiseCodexMaster`) or explicitly switch default mode.
2. Refactor Stage 1 to include **all** ordering-dependent functions (`generatePerFolderIndex`, `cascadeRegenerateIndexes`, `generateFolderHierarchy`) and explicit regeneration points.
3. Define a strict post-mutation pipeline: `mutate -> regenerate index cache -> reload tree` for index-mode edits.
4. Make stage docs self-contained (remove "copy from original" placeholders, include concrete snippets or exact file path anchors).
5. Update batch APIs so batch operations pass data directly (no nested prompt loops, no per-item redundant reload).
6. Preserve security patterns in all file/path snippets (workspace-root boundary checks, path sanitization).
7. Resolve command contribution hygiene in `package.json` (declare all menu-exposed commands, align IDs).
